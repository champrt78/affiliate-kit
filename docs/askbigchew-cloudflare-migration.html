Part 0Pre-flight check

Before starting, confirm you have these. None of them are surprises if you've already worked through BASEMENT_SETUP.md.

  • A Cloudflare account (free is fine), logged in at dash.cloudflare.com
  • The askbigchew.com login at namecheap.com
  • Push access to champrt78/askbigchew on GitHub
  • wrangler CLI authenticated (wrangler whoami shows your email)
  • The affiliate-kit link-cloaker Worker already deployed — see BASEMENT_SETUP.md Part 8 if not. Skip Part 7 of this guide until that's done.
Time budget Active hands-on time: ~30–45 minutes. Total wall time: 1–24 hours, gated by Namecheap → Cloudflare nameserver propagation. You can do Parts 4–6 while propagation finishes.

Part 1Add askbigchew.com to Cloudflare

  1. Log in to dash.cloudflare.com.

  2. Top-right + Add → Existing domain → type askbigchew.com → Continue.

  3. Plan picker: choose Free. Confirm. Cloudflare scans existing DNS records (there are basically none right now — Namecheap parking).

  4. You land on a "Nameservers" page that says something like:

    Replace your nameservers with:
  alice.ns.cloudflare.com
  bob.ns.cloudflare.com

    Your two nameservers will be different. Copy them — you need them in Part 2. Cloudflare also shows your Account ID in the right sidebar; that's already in ~/.claude/plugins/affiliate-kit/config.json from your earlier setup, so no need to copy it again.

  5. Leave this tab open. Don't click "Done, check nameservers" yet — we change Namecheap first.

Part 2Switch Namecheap nameservers

  1. Open a new tab. Log in to Namecheap → Domain List → next to askbigchew.com click Manage.

  2. On the Domain tab, scroll to the Nameservers section.

  3. Dropdown is probably set to "Namecheap BasicDNS". Change it to Custom DNS.

  4. Two text fields appear. Paste the two nameservers from Part 1 — one per field. Order doesn't matter.

  5. Click the green checkmark to save. Namecheap shows "Nameservers were successfully changed." You'll get an email confirming the change.

Heads up Namecheap may show a banner: "It may take up to 48 hours for the change to take effect." In practice it's usually 5–30 minutes for the major DNS resolvers to pick up the swap. Don't panic if it isn't instant.

Part 3Wait for propagation

Back in the Cloudflare tab, click Done, check nameservers. Cloudflare starts probing.

To check manually from this PC:

PowerShellnslookup -type=NS askbigchew.com 1.1.1.1

Once the answer comes back as your two *.ns.cloudflare.com entries (and not the old Namecheap ones), propagation is done.

Cloudflare will also email you "Cloudflare is now protecting askbigchew.com" — that's the green light.

While you wait Move to Parts 4–6. They don't depend on DNS being fully on Cloudflare yet — they prep the site code and the CF Pages project. You only need DNS finished before attaching the custom domain in Part 6.

Part 4Prep bc/ for static export

The site at C:\Users\Ray\Documents\GitHub\bc is Next.js 15 + MDX, filesystem-driven. It needs three small changes to build as a fully static site that Cloudflare Pages serves natively (no Workers/SSR runtime needed). All edits are in the bc/ repo, not affiliate-sites/.

4a — Enable static export

Edit bc/next.config.mjs:

bc/next.config.mjs/** @type {import('next').NextConfig} */
const nextConfig = {
  output: "export",                  // ← add
  trailingSlash: true,               // ← add (CF Pages likes /reviews/foo/ over /reviews/foo)
  pageExtensions: ["ts", "tsx", "mdx"],
  reactStrictMode: true,
  images: {
    unoptimized: true,               // ← add (Next image optimizer needs a server)
    dangerouslyAllowSVG: true,
    contentDispositionType: "attachment",
    contentSecurityPolicy: "default-src 'self'; script-src 'none'; sandbox;",
    remotePatterns: [
      { protocol: "https", hostname: "m.media-amazon.com" },
      { protocol: "https", hostname: "images-na.ssl-images-amazon.com" },
      { protocol: "https", hostname: "image.chewy.com" },
    ],
  },
};

4b — Pre-render dynamic routes

You have two dynamic routes: app/category/[slug]/page.tsx and app/reviews/[slug]/page.tsx. Static export needs each to declare its slugs at build time via generateStaticParams.

Reviews route — add this near the top of app/reviews/[slug]/page.tsx:

app/reviews/[slug]/page.tsximport { getAllReviewSlugs } from "@/lib/reviews";

export async function generateStaticParams() {
  return getAllReviewSlugs().map((slug) => ({ slug }));
}

Category route — same pattern, derived from the unique categories in lib/categories.ts:

app/category/[slug]/page.tsximport { categories } from "@/lib/categories";

export async function generateStaticParams() {
  return categories.map((c) => ({ slug: c.slug }));
}

The exact helper names (getAllReviewSlugs, categories) may need to be added to lib/reviews.ts / lib/categories.ts if they aren't exported yet. Quick sanity check:

PowerShellcd C:\Users\Ray\Documents\GitHub\bc
Get-Content lib\reviews.ts | Select-String "export "
Get-Content lib\categories.ts | Select-String "export "

4c — Verify it builds static

PowerShellcd C:\Users\Ray\Documents\GitHub\bc
npm run build

Success looks like a final line "Exported successfully" and an out/ directory containing index.html, reviews/<slug>/index.html for each review, etc. If the build complains about a route that needs a server, that page is using something incompatible with static export (cookies, dynamic params without generateStaticParams, server actions). Fix or remove before continuing.

Part 6Attach askbigchew.com as the custom domain

Requires Part 3 to be done — the zone must be active in Cloudflare before you can attach the apex.

  1. In the askbigchew Pages project → Custom domains → Set up a custom domain.

  2. Enter askbigchew.com. Cloudflare detects you own the zone and offers to create the DNS record automatically. Yes. It adds a CNAME (or a flat record on apex; CF handles the apex CNAME flattening for you).

  3. Repeat for the www subdomain: Set up a custom domain again → www.askbigchew.com. CF will auto-redirect www → apex using a Bulk Redirect rule it sets up for you.

  4. SSL certificate provisioning runs automatically in the background. Status shows "Pending certificate" then "Active" within 5–10 minutes. You don't have to do anything.

  5. Visit https://askbigchew.com. Should serve the site over HTTPS with a green padlock.

Part 7Wire up the link-cloaker

This part assumes the affkit-link-cloaker Worker is already deployed (per BASEMENT_SETUP.md Part 8). If not, do that first — come back here when wrangler deployments list --name affkit-link-cloaker shows a deployment.

7a — Attach the route

From the affiliate-sites/workers/link-cloaker/ directory, add a route entry to wrangler.toml:

workers/link-cloaker/wrangler.tomlroutes = [
  { pattern = "mywildlifecam.com/go/*", zone_name = "mywildlifecam.com" },
  { pattern = "askbigchew.com/go/*",    zone_name = "askbigchew.com" },   # ← add
]

Then redeploy:

PowerShellcd C:\Users\Ray\Documents\GitHub\affiliate-sites\workers\link-cloaker

Wrangler will create the Worker Route on the askbigchew.com zone for you. Verify in CF dashboard → askbigchew.com → Workers Routes: should show askbigchew.com/go/* → affkit-link-cloaker.

ReferenceTroubleshooting

"This site can't be reached" after nameserver swap

DNS hasn't fully propagated yet. Wait 15 minutes, then re-check with nslookup -type=NS askbigchew.com 1.1.1.1. If after 24 hours it's still pointing at Namecheap, the nameserver fields weren't saved correctly — go back to Part 2 step 5.

Cloudflare Pages build fails: "Cannot find module..."

Some path resolution differs between local Node and CF's build container. Usually fixed by setting NODE_VERSION = 20 in the project's environment variables. CF Pages defaults to 18 if not specified.

Cloudflare Pages build fails: "Page X is missing exports..."

A dynamic route doesn't have generateStaticParams. Find the offending file in the build log, add the function (see Part 4b for the pattern), push.

Custom domain shows "Verifying" forever

CF can't validate the domain because the zone isn't fully active yet. Wait until you've gotten the "Cloudflare is now protecting askbigchew.com" email, then refresh.

/go/<slug> returns the site's 404 page instead of the Worker's 404

The route isn't attached to the zone. CF dashboard → askbigchew.com → Workers Routes. If empty, re-run wrangler deploy from workers/link-cloaker/. If the route is there but Pages is intercepting, make sure the Worker route pattern askbigchew.com/go/* is listed above any Pages domain routing — Pages routes are catch-all by default.

HTTPS cert won't issue

Either DNS still hasn't fully propagated, or there's a CAA record at Namecheap restricting issuers. Check nslookup -type=CAA askbigchew.com. If there's a CAA record that doesn't include letsencrypt.org / pki.goog, edit it (or remove it) in Cloudflare's DNS panel — CF re-attempts every few hours.
