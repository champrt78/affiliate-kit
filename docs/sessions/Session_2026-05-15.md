### P1 — Proposed fixes / decisions (11 findings)

**Anchor 100 (evidence airtight, will hit in practice):**

1. **R9/R14 product-database contradiction (coherence + scope-guardian).** R9 says "a publishable piece cannot reference a product that has no verified database entry" — unconditional. R14 says "Piece #1 uses frontmatter-only specs (no product database yet)." These are mutually exclusive. Fix options: (A) add MVP carve-out to R9 ("except piece #1 per R14"); (B) build the database for piece #1 and drop R14's frontmatter-only carve-out.

2. **DRAFT gate is body-string-match, not section-name aware (feasibility, anchor 100).** The renderer at `templates/site-template/src/pages/reviews/[...slug].astro` uses `DRAFT_MARKER = "_Waiting for the human._"` as a literal string check. R6's `## Bottom Line` rename is a no-op for the gate UNLESS the MVP change set ALSO updates: (a) placeholder string in `templates/review.md.tmpl` + `templates/buyers-guide.md.tmpl`, (b) the `DRAFT_MARKER` constant in BOTH `[...slug].astro` renderers under `templates/site-template/`, (c) propagating those renderer changes into the 5 forked site copies under `sites/{mywildlifecam,fussybean,detailerpicks,starteraquarium,gameovergear}/`. Spans 7 .astro files + 2 .md.tmpl files. This is the single most load-bearing change in the MVP set and the doc does not call out the file-fork sync requirement.

3. **DRAFT banner UX entirely unspecified (design-lens).** AE2 says "page renders with a DRAFT banner and noindex meta tag" but no requirement defines banner content, placement, contrast, dismissibility, or whether it renders on the deployed URL or local-only. Two engineers would produce different banners and both satisfy R6.

4. **Amazon hotlink failure state undefined (design-lens + adversarial, cross-persona promoted).** R13 says hotlinks rotate quarterly but defines no fallback when an image 404s, no detection mechanism, and no acknowledgment of the Amazon Associates Operating Agreement position on hotlinking vs PA-API. By piece 5 across 5 sites, the probability of ≥1 broken hero image is non-trivial. Page positioned as "authoritative" with broken images undermines credibility.

5. **Reader value-add unvalidated but framework bets heavily on it (product-lens + adversarial, cross-persona promoted).** The anti-recipe-page + aggregation + use-case framing thesis is inferred from Ray's conviction, not validated by reader research. Dependencies section honestly admits this but treats it as a footnote. No kill-criterion. If pieces 1-5 underperform (low CTR, high bounce, no SERP ranking), there's no defined exit condition. SGE (Search Generative Experience) may also scrape the Bottom Line and serve it directly without sending the click — a real risk for answer-first pieces.

**Anchor 75 (will hit in practice, less airtight):**

6. **Buying-guide template ships explicit hands-on disclaimer that contradicts R11 (feasibility).** `templates/buyers-guide.md.tmpl` opens with "**Note:** This is a buyer's guide, not a review. We haven't tested this product ourselves." and the Editor's Note section says "I haven't owned and tested __PRODUCT_NAME__ personally." Scope Boundaries explicitly REJECTS this framing. MVP minimum-change set must delete/rewrite these lines; doc does not call this out.

7. **R5/R6 gating ambiguity for buying-guide pieces (coherence).** R5 places `## Bottom Line` at top with no piece-type qualification; R6 makes it the DRAFT gate. Buying-guides may have different structural needs (use-case frame, category overview) — does the same gate apply uniformly? AE2/AE3 are ambiguous on piece type.

8. **R2 voice doctrine document location/version not specified (coherence).** Where does the file live? What format (YAML/JSON/markdown)? How is it versioned? Implementers cannot write the scaffolding-script prompt-injection code without this. Deferred to planning per the doc's Outstanding Questions, but worth being explicit.

9. **Bottom Line placeholder-detection mechanism undefined (design-lens).** R6 says "the page will not index until the placeholder is replaced with the publisher's own prose." Detection contract is unspecified — sentinel string? Empty heading? Character-length threshold? Frontmatter flag? Two implementers would pick different mechanisms; gate behaves inconsistently. Recommended: scaffold inserts a unique sentinel string (e.g., `BOTTOM_LINE_PLACEHOLDER`) the renderer checks for explicitly.

10. **Voice doctrine forbids generation-time phrases but doesn't prescribe response to direct confrontation (adversarial).** Concrete failure scenario: reader DMs "did you actually test this?" or FTC investigator emails the site asking the same. Silence reads as concealment; an improvised "yeah we tried it" collapses the entire legal-defense pillar. Recommend: add a "direct-question response template" to the voice doctrine — canonical doctrine-aligned responses for reader/manufacturer/regulator queries — drafted BEFORE piece #1 ships.

11. **Per-site product database has no staleness/refresh requirement (adversarial).** R8 stores verified-date; R9 forbids unverified product references. Neither requirement defines when a "verified" entry expires. By piece 8-10 with overlapping products, stale specs leak silently while the verified-date implies currency — worse than no verification at all. Recommend: bind verified-date to a staleness window (180 days surfaces a re-verify warning; 365 blocks piece generation) tied to the existing 90-day cycle.

15. **R3 AI prompt construction infrastructure doesn't exist (feasibility).** Current `scripts/new-review.ps1` and `scripts/buyers-guide.ps1` are pure string-substitution templating; they never invoke an AI. R3 implies a new capability (API client? agent-side conventions file? plugin-command wrapper?). Magnitude of the gap is not surfaced. 16. **Goal-work misalignment: "passive income" vs publisher-on-critical-path (product-lens, error).** Publisher writes every Bottom Line, verifies every spec, reviews every draft. The most load-bearing step (Bottom Line synthesis gating the DRAFT) is explicitly human-authored to make the legal defense work. "Passive" is misleading framing. Reframe as "AI-leveraged at publisher-sustainable cadence" OR add a Key Decision acknowledging the trade.
17. **Reader-segment fit asserted but not traced through framework (product-lens).** R12 names mywildlifecam segments (homeowners/property owners/first-timers/backpackers; NOT hunters). No requirement binds AI output to those segments. AI default may converge on hunters (dominant trail-cam search audience). Recommend: inject site primary/secondary segments + exclusions into the AI prompt as a constraint.
18. **Buying-guide comparison table rendering pattern undefined (design-lens).** R5 lists "comparison sections" with no rendering spec. Table vs cards, required columns, mobile behavior, recommended-pick visual treatment — all unspecified. This is the primary conversion surface for multi-product queries. Without a pattern, implementers default to a generic equal-weight table — exactly the AI-slop failure mode.
19. **DRAFT gate is presence check, not quality check (adversarial).** Publisher under deadline writes "This is a solid choice for most buyers." as Bottom Line — one banal sentence — gate clears, page indexes, legal-defense pillar silently degrades. Worse: publisher pastes AI-generated text into Bottom Line. The gate cannot detect authorship origin. Either (a) document the gap explicitly ("the gate enforces human presence, not quality"), or (b) add a lightweight quality signal — min word count, frontmatter `bottomline: human` assertion, presence of at least one product-name reference.
20. **MVP-first rollout has no defined rollback/revision path (adversarial).** F3 covers GROWING the system but not REVISING MVP decisions. What if piece #3 reveals the section rename was wrong, or `## Who This Is For` reads as filler? Voice doctrine evolution: pieces drafted under v1 may have phrases that become forbidden by v2 — no back-check defined. Add a "Framework revision policy" — when doctrine evolves, prior pieces flagged for review at quarterly refresh; when anatomy changes, define migrate-vs-grandfather; name the trigger for piece-level → framework-level escalation (e.g., "3 pieces in a row hit the same edge case → doctrine update").
