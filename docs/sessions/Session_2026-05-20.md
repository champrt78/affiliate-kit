## Discoveries / Gotchas

- **CSS `aspect-ratio` does NOT prevent CLS on its own.** Even when the parent has `aspect-ratio: 21/9` and the `<img>` has `width:100%; height:100%; object-fit:cover` in CSS, the browser still reserves 0 height during initial HTML parse if the `<img>` element lacks `width`/`height` attributes. The CSS only kicks in once the stylesheet is applied, which on a brand-new domain with cold cache leaves a measurable shift window. HTML width/height attrs are the only way to give the browser an aspect-ratio hint *during parse*. Rule of thumb across the repo: every `<img>` gets width+height attrs even if CSS already constrains the container. Documented in CLAUDE.md.
- **Cloudflare Web Analytics is bot-filtered; the Traffic panel is not.** Domain-level traffic counts on the account overview (1.1K, 580, etc.) are raw edge requests including all bot traffic. For human-vs-bot signal, use the per-zone **Analytics & Logs → Web Analytics** tab, which has to be enabled per zone. Free, works without a JS tag when the zone is proxied (orange-cloud DNS).
- **Cloudflare's "Enable all available settings" button is the right portfolio tool, not a per-zone audit.** Each zone exposes a Site Recommendations card on the Speed → Settings → Recommendations page with a one-click button that flips every Free-tier-safe setting that's currently off. Saves clicking through Speed Brain / 0-RTT / Always Use HTTPS / Early Hints individually. Pro/Business/ACM-gated settings (Polish, Enhanced HTTP/2 Prioritization, Advanced Certificate Manager, Total TLS, Cipher Suites) stay off because they're not available on Free.
- **Cloudflare dashboard load can be slow on first hit to a less-visited zone** (spacemonkeylabs.space, starwatchstation.space both spent 10–15s on the orange-cloud splash before rendering Speed settings). Don't conclude the page is broken — just wait. Once rendered, the UI is fast.
- **HSTS is a one-way door — don't enable it casually.** Even on Cloudflare's UI it's a single button, but the *consequence* persists in every visitor's browser for `max-age` seconds. If TLS breaks for any reason during that window, those visitors get hard-blocked. For new domains specifically, ship with `max-age=300` (5 min) for a few days to confirm nothing breaks, then ramp to 6mo+. Never enable `preload` unless you're sure you'll never want HTTP back.
- **Lockfile drift on a fresh clone produces a misleading Vite error.** First build attempt on the home PC failed with "Rollup failed to resolve import `@affkit/shared-utils`" — looked like a real bug in `ReviewArticle.astro` but was just stale `node_modules` vs an updated lockfile. `pnpm install` fixed it. Vite's "Re-optimizing dependencies because lockfile has changed" line is the actual signal.
- **Amazon PA API rule has tightened.** The widely-circulated "3 sales in 180 days" requirement is outdated. As of the current associates.amazon.com page: **10 qualifying sales in trailing 30 days, ongoing.** This is a maintenance bar, not just an activation gate — drop below and access lapses. Documented here so future-Ray doesn't waste time on PA API plans.
- **Amazon Creators API pathway** exists as a separate route to PA API access via the Amazon Influencer Program. Not pursued — requires active social audience (YouTube/TikTok/Instagram) which doesn't match the blog-site portfolio.
- **Canopy requires a card on file even for the Hobby tier.** Standard anti-abuse pattern (same as Vercel/Cloudflare Workers/Supabase). $0 auth, no real charge. Hobby's hard request cap means runaway scripts can't trigger overage billing — only an explicit plan switch could.

### Discoveries / Gotchas

- **`:has()` selector is the load-bearing piece** for the "whole-paragraph standalone `<strong>`" treatment. Astro supports it natively in `<style>` blocks; no PostCSS plumbing needed. All major browsers ship it as of 2024+ (Baseline). If a target browser doesn't support `:has()` the rule no-ops gracefully (the inline-bold still renders, just without the pull-quote treatment) — acceptable fallback.
- **Body-bg gutters are essentially invisible until viewport > 1320px.** On a 1280px laptop screen Ray won't see the gutters at all (the shell consumes the viewport). The mid-width breakpoint (`max-width: calc(100vw - 48px)` between 760px and 1380px) gives a 24px gutter at typical laptop widths so the editorial feel is still present even when the viewport is narrower than the shell's 1320px cap.
- **The page-shell drop shadow (`0 24px 60px -16px rgba(15, 26, 21, 0.35)`) is what makes the cream card read as "floating on the table" rather than "wallpaper behind the content."** Without it the gutter still works visually but feels flat. The shadow is the difference between "background color" and "design intent."
- **Stripped inline `transform: scale(imageScale)` is now used only on the in-article hero** (`ReviewArticle.astro`), not on cards or per-pick deep-cards. The hero is one product per page so imageScale doesn't cause cross-product mismatch — it's a per-product fine-tune for whitespace in the source PNG. Document convention: imageScale lives on hero, never on lists.

### Iteration 2 — ReviewCard image-size normalization (Task #21 finish)

**The remaining gap from iteration 1 finally closed.** `packages/shared-ui/src/components/ReviewCard.astro` was rendering products at wildly different sizes on `/reviews/` and the homepage's "Latest reviews" row. Root cause was the combo of `max-width: 65%` (loose width-only constraint) + per-product `imageScale` `transform: scale(...)` (different value per product, fragile).

Removed `transform: scale()` application on the card layer entirely (still in use on ReviewArticle hero — different use case, different file).

**After:** Spypoint 265×164, Tactacam 207×207, Stealth Cam 207×193 (fixed height 78% → all heights within ~25px of each other; widths differ because actual products differ in real-world aspect, which is correct).
