### Late evening: the 6-commit gutter-debugging saga

Body-bg gutters were invisible on both sites despite the per-page theme commits ostensibly landing. Each fix was real but addressed a layer that wasn't the root cause:

- `1e54129` — Injected body bg via `<style is:inline>` in BaseLayout's head slot. (Why: Astro's `define:vars` scopes `--pageBg` to the component's root element + descendants; `:global(body)`'s var() lookup couldn't see it because vars only cascade DOWN.)
- `dfecef0` — Removed the now-redundant compiled `body { background: var(--pageBg) }` rule from MainLayout. (Why: compiled CSS file loads AFTER the inline head style; without removing this, the broken var() reference clobbered the inline rule and unresolved var fell to invalid → no bg.)

- `6acbabb` — **`main { background: transparent }`** added. **THIS was the actual root cause**. `<main>` had a cream `var(--color-bg)` painting on top of body — body bg was always rendering correctly, `<main>` was covering it horizontally. Photo only escaped in the vertical gap between page-shell bottom and footer top.
- `f07caf0` — **Deleted a DUPLICATE `main` rule in DTP layout** that survived the previous fix. Two `main[data-astro-cid-ouamjn2i]` rules with the same selector, same specificity; the later (cream) one won the cascade. ## Discoveries / Gotchas

- **Amazon `/images/P/<ASIN>.01.LZZZZZZZ.jpg` returns aggressive close-up crops, NOT the canonical product image.** This is an old thumbnail-API URL pattern. Use Canopy's `mainImageUrl` (`m.media-amazon.com/images/I/<id>.jpg`) instead — that's the full-product shot. Document this in the content workflow: any new product card pulls image via Canopy, never `/images/P/...`.
- **Canopy API is the right default for product data.** ASIN in → title + price.display + mainImageUrl + rating + ratingsTotal + featureBullets out. Free tier (Hobby plan, 100 req/mo) is sufficient for current content volume. Used live for the first time today on Mr. Pink + GardePro. Should replace all manual browser-tab paste-ins going forward.
- **Voice-lint catches "I own" / first-person possessive even in QUOTED owner-review strings.** When paraphrasing forum reviews, write "safe-on-every-wheel-type confidence" rather than quoting "safe on every wheel I own."
- **AvantLink's verifier does literal-string-match on the URL it generated.** Switching `http://` → `https://` for mixed-content compatibility makes verification fail. Their `affiliate_app_confirm.php?mode=js` script doesn't need to execute, just be present in HTML source — so keep it on the http:// form they sent.
- **CF Pages custom-domain `mywildlifecam.com` apex can serve stale content** even after a successful production deploy. Verified via wrangler that the project's production deploy is correct + pages.dev URL returns the new content + cf-cache-status is DYNAMIC, but the apex still ships old HTML. Manual cache purge from CF dashboard didn't unstick it either. Logged as the AvantLink verification blocker; needs deeper investigation.
- **Astro `define:vars` scopes CSS custom properties to the component's root element + its descendants.** A `:global(body) { background: var(--pageBg) }` rule will silently fail because body is NOT a descendant of the component's root — it's the ancestor. CSS custom properties cascade DOWN only.
- **CSS `background:` shorthand cannot have a bare color as its own comma-separated layer.** `linear-gradient(...), url(...), #14181C` is invalid syntax; entire declaration is rejected. The color must either attach inline to the last layer (no comma) OR be a separate `background-color:` declaration AFTER the shorthand (since shorthand resets bg-color to transparent).
- **Both site layouts had a leftover `main { background: var(--color-bg) }` rule that covered the body bg horizontally.** Removing it was the actual fix to the per-page gutter visibility. DTP had it TWICE in the same style block — same selector, same specificity, the later one won; verified via chrome-devtools `matchedCSSRules`.
- **Lesson on debugging workflow:** the gutter saga took 6 fix commits. I chased CSS-shorthand bugs and var-scoping issues in source code before inspecting the live DOM. Chrome DevTools MCP + a 5-line `getComputedStyle` script would have shown the cream `<main>` background on the first attempt and saved 4-5 round-trips. **Rule for future sessions:** at the FIRST report of a visual bug, navigate to the page via chrome-devtools-mcp and run `getComputedStyle` on the affected element BEFORE editing CSS source.
- **/scaffold-piece's wrapper has an arg-parsing bug** that triggers when the new-review.ps1 script passes args to add-link.ps1 — it sends `-Merchant -Url` together instead of as separate args. Workaround: register the cloaker KV manually with `pwsh scripts/add-link.ps1 -Site <slug> -Slug <slug> -Url <url> -Tag amazon`. Worth fixing in the wrapper.
